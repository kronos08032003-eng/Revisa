1 — Linux: do básico ao avançado

Comandos práticos, padrões de uso, administração de arquivos, permissões, processos, rede e scripts shell.

1.1 Navegação e arquivos

  • pwd — mostra diretório atual.
  • ls -la --color=auto — lista com detalhes; cores se disponíveis.
  • cd - — volta para o diretório anterior.
  • find . -type f -name '*.log' -mtime -7 — arquivos .log modificados nos últimos 7 dias.
  • du -sh pasta/ — tamanho da pasta (resumido).

1.2 Permissões e propriedade

  • ls -l arquivo — mostra permissões (rwx) e dono/grupo.
  • chmod 644 arquivo — dono rw, grupo/leitores somente leitura.
  • chmod +x script.sh — torna executável.
  • chown usuário:grupo arquivo — altera dono e grupo.
  • Use getfacl / setfacl para ACLs mais granulares (se disponível).

1.3 Processos e monitoração

  • ps aux | grep nome — procura processo.
  • top / htop — monitor em tempo real.
  • systemctl status nginx — status de serviços (systemd).
  • journalctl -u nome-servico -f — logs do systemd (seguimento).
  • nice / renice — ajustar prioridade.

1.4 Rede

  • ip addr show — interfaces e IPs.
  • ss -tuln — sockets/listeners.
  • curl -I https://dominio — checar cabeçalhos HTTP.
  • tcpdump -i eth0 port 80 — captura de pacotes (precisa sudo).
  • dig +short @8.8.8.8 exemplo.com — consulta DNS direto a um resolver.

1.5 Scripts e boas práticas

Padrão para scripts robustos:

#!/usr/bin/env bash
set -euo pipefail
IFS=$'\n\t'

# Exemplo: processar .csv
for f in *.csv; do
  echo "Processando $f"
  # comando de processamento
done
  • Use set -euo pipefail para falhas detectáveis.
  • Redirecione saída e erros: cmd > out.log 2>&1.
  • Documente com cabeçalho e opções (getopts / longopts).
Tip: para testes locais de servidor estático, python3 -m http.server 8000 (na pasta do site) — acessa em http://localhost:8000.

2 — Git: do básico ao avançado (fluxos reais)

Conceitos, comandos e estratégias (feature branches, release, rebase, merge, cherry-pick, hooks e resolução de conflitos).

2.1 Configuração e fluxo básico

$ git config --global user.name "Seu Nome"
$ git config --global user.email "seu@exemplo.com"
$ git init
$ git remote add origin git@github.com:usuario/repo.git
$ git add .
$ git commit -m "chore: iniciar repositório"
$ git push -u origin main

Dica: configure git config --global core.editor para editar commits com seu editor preferido.

2.2 Branching e estratégias

  • Git Flow: main, develop, release/x.y, feature/x, hotfix/x — usado em times maiores (mais processos).
  • Trunk-based: branches curtos, merge frequentemente para trunk; ótimo para CI/CD.
  • Use git checkout -b feature/x para criar e mudar para uma branch.

2.3 Rebase vs Merge

Merge preserva histórico (cria commit de merge). Rebase reescreve commits locais para aplicar em cima de outra base — deixa histórico linear.

Quando usarComando
Manter histórico íntegrogit merge main
Linearizar commits antes de pushgit rebase main

2.4 Resolução de conflitos

  1. Executar git status e identificar arquivos em conflito.
  2. Abrir e editar removendo marcadores <<<<<<.
  3. git add arquivo e git commit (ou git rebase --continue).

Ferramentas: VSCode, meld, kdiff3, Beyond Compare ajudam na visualização.

2.5 Hooks, stash e cherry-pick

  • git stash → guardar mudanças locais temporariamente.
  • git cherry-pick <commit> → aplicar commit específico em outra branch.
  • Hooks locais (pasta .git/hooks): pre-commit, pre-push para lint/format/checks.

2.6 Inspeção de histórico

$ git log --oneline --graph --decorate --all
$ git blame arquivo --line-por-line
$ git bisect start

3 — Plataformas de hospedagem estática (comparação detalhada)

Avaliação aprofundada entre GitHub Pages, Netlify, Vercel, Cloudflare Pages e AWS S3 + CloudFront — critérios: facilidade, integração Git, HTTPS, CDN, funções edge, custo e limites.

3.1 Resumo comparativo (visão geral)

Plataforma Integração Git HTTPS Automático Funções / Edge Use quando
GitHub PagesPush → deploy (GitHub)Sim (Let's Encrypt)—Doc, portfólio, sites simples
NetlifyCI integrado (GitHub/GitLab/Bitbucket)SimFunctions (serverless)Sites JAMstack, deploy simples
VercelIntegração profunda com GitSimEdge Functions (serverless)Next.js, apps React estáticos/dinâmicos
Cloudflare PagesGit + CISimWorkers (muito rápido na borda)Performance global, apps edge
AWS S3 + CloudFrontCI externo (ex: GitHub Actions)Sim (ACM)Lambda@Edge (avançado)Escala empresarial, controles finos

3.2 Custos e limites (resumo)

  • GitHub Pages: totalmente gratuito para repositórios públicos.
  • Netlify: conta grátis generosa; limites em builds/requests no plano gratuito.
  • Vercel: grátis para uso pessoal e projetos pequenos; limites em builds e execução de funções.
  • Cloudflare: plano gratuito com limites altos; excelente CDN global.
  • AWS: custo pay-as-you-go; barato em escala pequena, mas precisa gerenciar recursos.

3.3 Quando escolher cada uma

  • Escolha GitHub Pages para documentações e sites sem funções.
  • Netlify se quiser funções serverless simples + formulários e integração rápida.
  • Vercel para projetos Next.js e necessidade de Server-Side Rendering (SSR) ou Edge Routes.
  • Cloudflare Pages para latência global e workloads em Workers.
  • AWS S3 + CloudFront para controle máximo e compliance corporativa.

3.4 Exemplo prático: deploy com GitHub Pages (passos)

# 1. Criar repositório no GitHub (public)
# 2. No repositório: Settings → Pages → selecionar branch (ex: main) e pasta (/root ou /docs)
# 3. Push do código
# 4. Aguardar build; seu site ficará em https://usuario.github.io/repositorio
Observação: plataformas como Netlify/Vercel geram previews por pull request — útil para revisão antes do deploy em produção.

4 — HTTPS: funcionamento técnico e certificados (ACME / Let's Encrypt)

Explicação passo a passo de como HTTPS funciona, negociação TLS, certificados X.509 e automação via ACME.

4.1 Como funciona um handshake TLS (simplificado)

  1. ClientHello — o cliente envia versões e ciphers suportados.
  2. ServerHello — o servidor escolhe cipher e envia certificado X.509.
  3. Validação do certificado — cliente verifica cadeia de confiança (CA), validade e nome (CN/SAN).
  4. Troca de chaves — geralmente ECDHE para criar chave de sessão efêmera.
  5. Comunicação simétrica cifrada (AES/GCM, etc) usando a chave de sessão.

TLS protege confidencialidade, integridade e autenticidade. Versões seguras atuais: TLS 1.2 e TLS 1.3 (preferir 1.3 quando possível).

4.2 Certificados e ACME (Let's Encrypt)

Let's Encrypt é uma CA gratuita que usa o protocolo ACME para validar domínios e emitir certificados. Ferramentas populares: certbot, acme.sh.

# Exemplo rápido com certbot (certbot precisa estar instalado)
sudo certbot certonly --standalone -d exemplo.com -d www.exemplo.com

# Renovações automáticas (certbot instala um cronjob/systemd timer)
sudo certbot renew --dry-run

Muitos provedores (Netlify, Vercel, Cloudflare) emitem e renovam certificados automaticamente — sem intervenção manual.

4.3 Cabeçalhos de segurança recomendados

  • Strict-Transport-Security (HSTS) — forçar HTTPS: max-age=63072000; includeSubDomains; preload.
  • Content-Security-Policy — reduzir riscos de XSS.
  • Referrer-Policy, X-Frame-Options, X-Content-Type-Options.

Tenha cuidado com HSTS em domínios que ainda servem conteúdo via HTTP — uma vez aplicado por longos períodos pode causar bloqueios.

5 — DNS: conceitos, registros e exemplos práticos

Como funcionam zonas DNS, principais registros (A, AAAA, CNAME, TXT, MX, NS) e exemplos de configuração para hospedagem estática.

5.1 Registros DNS principais

  • A — aponta nome para IPv4 (ex: exemplo.com → 93.184.216.34).
  • AAAA — aponta para IPv6.
  • CNAME — alias para outro nome (não usar para raiz/apex).
  • MX — servidores de e-mail.
  • TXT — verificações, SPF, DKIM, etc.
  • NS — delega autoridade da zona a nameservers.

5.2 Exemplos práticos de records

Exemplo: apontar domínio para Netlify (CNAME para your-site.netlify.app) e apex para registros A fornecidos pelo provedor:

# Apex (exemplo)
exemplo.com.    300  IN  A   104.198.14.52

# Subdomínio www
www.exemplo.com. 300 IN CNAME your-site.netlify.app.

# Verificação (TXT)
_acme-challenge.exemplo.com. 300 IN TXT "token-de-validacao"

Ferramenta de diagnóstico: dig +trace exemplo.com e dig @8.8.8.8 exemplo.com ANY.

5.3 TTL e propagação

TTL (time-to-live) é o tempo que resolvers cacheiam uma entrada. Ao alterar records, pode demorar até o TTL para a mudança propagar. Para deploys críticos, reduza o TTL antes da mudança (ex: 300s), e volte após estabilizar.

6 — Workflows e Deploys passo-a-passo

Guias práticos para publicar sites em diferentes provedores, incluindo DNS e HTTPS.

6.1 Deploy rápido: GitHub Pages

# Local (exemplo)
git init
git add .
git commit -m "site inicial"
git branch -M main
git remote add origin git@github.com:usuario/repositorio.git
git push -u origin main

# No GitHub: Settings → Pages → source: main branch → root (/)
# URL: https://usuario.github.io/repositorio

6.2 Deploy com Netlify (CI automático)

  1. Criar conta Netlify e conectar ao repositório (GitHub/GitLab/Bitbucket).
  2. Configurar build command (ex: npm run build) e publish directory (ex: dist).
  3. Adicionar domínio customizado no painel Netlify; Netlify orienta a configuração DNS.
  4. HTTPS é emitido automaticamente por Let's Encrypt.

6.3 Deploy com Vercel

  1. Conectar repositório no Vercel.
  2. Vercel detecta framework e configura build (Next.js, Gatsby, etc.).
  3. Preview URLs são gerados por pull request.
  4. Adicionar domínio e apontar DNS conforme instruções (CNAME/A ou Cloudflare integration).

6.4 Exemplo: Configurar DNS para Cloudflare + Pages

  1. Adicionar domínio ao Cloudflare e apontar nameservers para Cloudflare (no registrador).
  2. No Cloudflare Pages, conectar repositório e configurar build.
  3. O Cloudflare cuida de certificados TLS e CDN; ative Always Use HTTPS e HSTS conforme necessidade.

7 — Troubleshooting & Erros comuns

Lista de problemas típicos e como diagnosticar rapidamente.

7.1 Problemas com HTTPS

  • Certificado inválido → verifique data/hora da máquina e registros DNS.
  • Erro de nome (CN/SAN) → certificado não cobre o subdomínio solicitado.
  • Renovação falhou → checar ACME challenge (TXT/CNAME) e permissões do servidor.

7.2 Problemas com DNS

  • DNS antigo em cache → usar dig +trace e limpar cache local/ISP.
  • CNAME no apex → muitos registradores não permitem; use ALIAS/ANAME ou registros A.
  • Erro 404 após apontar domínio → verifique publish directory no provedor (ex: /public, /dist).

7.3 Debug de deploys CI

  • Logs de build: revisar console de build do provedor (Netlify/Vercel/GitHub Actions).
  • Ambiente: variáveis de ambiente (API keys) podem faltar no provedor; configure no painel.
  • Falhas de routing (SPA): habilitar redirects (ex: /_redirects no Netlify) para servir index.html.
